# Agent Handoff: Shieldbearer Website

System knowledge and rules for any human or AI agent working on
this repository. Read this file fully before taking action. The
companion file `AGENT_STATE.md` carries the live working state
(current branch, next task, execution mode); `MEMORY.md` is the
entry-point index that points at every doc in this project.

This file is the answer to "what is this project, how does it
work, what are the rules I must not break."

## What this project is

`shieldbearer-website` is the official artist website for
Shieldbearer (also written "Shield Bearer"), an independent
Christian metal recording project run by Moncy Abraham. The site
is the public surface for the music, the press kit, the
theological writing, and the direct contact path. It is hosted
on GitHub Pages from this repository on the
`sentinelbot-stable` branch. The production URL is
`https://shieldbearerusa.com`.

This is a real production site, not a draft. Every change
deploys to live within minutes of pushing to
`sentinelbot-stable`.

## Stack and structure

- Static HTML, CSS, JavaScript. No build step. No framework.
- 25 pages at the repo root, each as a `.html` file.
- Each `.html` file has a clean-URL mirror at `<page>/index.html`
  (for example `music.html` and `music/index.html`). Both must
  stay byte-identical (one documented exception: `sentinelbot`).
- All CSS is in `css/style.css`.
- JavaScript is in `js/` (main.js, featured-release.js,
  song-meanings-augment.js, merch-rotator.js, sentinelbot.js,
  signal-countdown.js, analytics.js).
- Live site data is in `site.json` at the repo root, generated
  by the SentinelBot publisher Lambda from a separate repo.
- Merch data is in `data/merch.json`, generated by a Shopify
  scrape script.
- Assets in `images/`. Audio (when present) in `audio/`.

## Standing rules (non-negotiable)

These are caught by `scripts/test.sh` automatically. They are
also project conventions a human reviewer will reject changes
over.

1. **No em dashes anywhere.** Hard rule across all source files,
   docs, commit messages, and human responses. Use commas,
   periods, colons, or parentheses. The em-dash guard at test
   step 21 scans for three forms of the em-dash character: the
   literal Unicode codepoint U+2014, the named HTML entity for
   it, and the backslash-u JS escape for the same codepoint.
   Per-line whitelist sigil
   `em-dash-allow` exists for legitimate uses (CSS pseudo-content
   typographic separators and intentional regex strips). See
   commits `eac2b36` and the discussion of legitimate uses in
   `SEO.md`.
2. **CHANGELOG is a gate, not paperwork.** Every change must add
   a versioned entry to `SHIELDBEARER_WEBSITE_CHANGELOG.md`.
   Versioning rules: patch for fixes, minor for visible features,
   major for architecture changes. Skipping the changelog is the
   most common slip; treat it as a release blocker.
3. **Mirror parity.** Every edit to `<page>.html` must be
   propagated to `<page>/index.html` via `cp`. Verified by test
   step 24 on every test run. Sentinelbot is the documented
   exception; nothing else may diverge.
4. **No same-href-same-label duplicates in nav surfaces.** Test
   step 22 catches duplicate links in dropdowns and the mobile
   menu. Different labels for the same href is allowed (Music
   and The Armory both pointing at /music is intentional UX).
5. **og:url matches canonical.** Test step 23 enforces this so
   social shares and Google's canonical signal point to the same
   URL form.
6. **External target=_blank links must carry rel=noopener.**
   Test step 27 catches tabnabbing vectors.
7. **Title uniqueness across pages.** Test step 26.
8. **No duplicate IDs within a page.** Test step 25.

## Test gate

`scripts/test.sh` runs:

- 27 structural HTML and metadata checks across 25 pages
  (198 total assertions when all pass)
- 53 jsdom regression tests on the four reader scripts
  (featured-release, song-meanings-augment, merch-rotator,
  signal-countdown)
- A 90% lines + statements coverage gate on those scripts via
  c8

Run `./scripts/test.sh` before every commit. The exit status is
the gate. Hooks block deploy on failure.

## Branch model and deployment

- `sentinelbot-stable` is the production branch. GitHub Pages
  serves from it. A push triggers a Pages rebuild within roughly
  1 to 3 minutes.
- Feature branches (e.g. `seo-fixes-2026-05`) are created from
  `sentinelbot-stable`, work in isolation, and either merge back
  via fast-forward (preferred, preserves atomic commits) or via
  a merge commit if linear history is impossible.
- Auto-published commits from the SentinelBot publisher
  ("auto-update: SentinelBot release sync ...") arrive on
  `sentinelbot-stable` periodically. Pull before branching to
  avoid drift.
- Hot-fix pattern: commit on `sentinelbot-stable`, push, wait
  for Pages rebuild (use `gh api repos/.../pages/builds/latest`
  to confirm the new commit deployed), then verify live with
  `curl`. Tell the operator when the change is visible.

## Commit conventions

- Atomic commits per task. Multiple small changes are squashed
  only when they are genuinely a single logical unit.
- Commit messages explain the why and the what, not just the
  diff. Reference prior commits by short SHA when relevant.
- Co-authored-by trailers only when explicitly requested.
- Never push to `sentinelbot-stable` from a feature branch
  without confirming with the operator first.

## Common failure modes (avoid these)

1. Editing `<page>.html` without copying to `<page>/index.html`.
   Caught by test 24 but easy to miss before run.
2. Skipping the CHANGELOG entry on a change. Caught by code
   review only, not by the test gate.
3. Inventing facts about the artist or the project in
   audience-facing copy. Always ask the operator before writing
   assertions about live capability, schedule, identity, or
   technical setup.
4. Using em dashes in chat replies and human prose even when
   the source files are clean. The hard rule covers all
   communication, not just code.
5. Pushing to `sentinelbot-stable` without operator confirmation.
6. Treating the auto-published `site.json` commits as a base to
   rebase onto without pulling first.
